# Proofly Technical Stack
## Real-only architecture for the wallet, ZK proof layer, and app backend

This document defines the exact stack for Proofly as a buildable system with one chain, one wallet extension, one operational database, and one proof layer.

The design principle is simple:

- trust-critical state lives on-chain,
- user keys and approvals live in the wallet extension,
- operational app data lives in Supabase,
- proofs are generated by a real ZK provider,
- nothing important is mocked.

---

## 1. Stack decision

### Chain
**Base Sepolia only**

Why this chain:
- it is EVM-compatible,
- it has official public testnet RPC endpoints,
- it is lightweight enough for a hackathon,
- it keeps the contract layer simple,
- it avoids multi-chain fragmentation.

Chain details to hardcode:
- Chain ID: `84532`
- Network: `Base Sepolia`
- RPC: `https://sepolia.base.org`
- Explorer: Base Sepolia explorer from the official Base docs

The app should not support any other chain in v1.

### Wallet
**Chrome Extension Manifest V3 wallet**

The wallet is the primary product surface. It must:
- generate or import a keypair locally,
- store encrypted private key material locally,
- expose an EIP-1193 compatible provider,
- support message signing,
- support typed data signing,
- support transaction signing,
- support chain switching,
- launch the ZK verification flow,
- enforce AI agent permissions.

### ZK proof provider
**World ID / IDKit**

Use World ID for proof-of-human and anti-bot verification.
Use the wallet to store the resulting proof state and nullifier usage.

### Database
**Supabase**

Use Supabase for:
- Postgres tables,
- Realtime updates,
- Auth if needed for admin or dashboard access,
- Edge Functions for proof verification relay and webhook handlers,
- Storage for media or audit attachments,
- Row Level Security for app-level authorization.

### Smart contracts
**Solidity on Base Sepolia**

Use contracts for:
- proof registry,
- policy registry,
- session or delegation limits,
- task gating,
- audit records where public on-chain state is needed.

### Frontend
**Next.js**

Use Next.js for:
- the public dApp,
- the worker marketplace,
- the AI agent console,
- the admin dashboard if needed.

### Shared tooling
- TypeScript
- viem or ethers.js for client-chain interaction
- Prisma only if you want an extra ORM layer over Supabase Postgres
- Tailwind CSS for UI
- shadcn/ui for fast components

## 2. Why this stack is the right shape

Proofly needs three different kinds of truth:

1. **Private truth**  
   Wallet keys, approvals, and local proof state.

2. **Public truth**  
   On-chain verification, policy hashes, and trust records.

3. **Operational truth**  
   Dashboard data, task records, media records, notifications, and indexed audit events.

That is why the stack is split between:
- wallet extension,
- Base Sepolia contracts,
- Supabase backend.

This is the cleanest version of “real, end-to-end, no mocks.”

## 4. Browser security and extension rules

The extension must follow Manifest V3 constraints:
- background logic runs as a service worker,
- remote code must not be loaded into the extension,
- content scripts run in the page context boundary,
- permissions should stay minimal,
- the extension package must contain the service worker code.

The wallet should not depend on remote script injection or remote “wallet logic” hosted on a server.

## 5. ZK layer

### Primary proof system
World ID / IDKit.

### What the proof does
- proves a real human is behind the wallet action,
- provides a nullifier per action scope,
- prevents the same proof from being reused where it should not be reused,
- allows dApps and contracts to treat the user as verified human without exposing identity.

### What to store
- proof provider name,
- action name,
- nullifier hash,
- verification timestamp,
- verification status,
- wallet address,
- chain binding if the proof is used on-chain.

### What not to store
- raw biometric data,
- raw proof internals if not needed,
- any identity data that is not necessary for the app flow.

## 6. Supabase layer

Supabase is the operational backend only. It is not the trust source.

### Why Supabase
Supabase gives:
- Postgres,
- Realtime,
- Auth,
- Storage,
- Edge Functions,
- row-level authorization.

### How it fits Proofly
Use Supabase for:
- wallet metadata index,
- proof event logs,
- task submissions,
- AI agent policy records,
- media hashes,
- off-chain attachments,
- dashboard views,
- notifications,
- indexing on-chain events.

### Security model
- turn on RLS for every table,
- use JWT-based session rules,
- expose only the rows the current user may read or write,
- keep privileged writes inside Edge Functions or server-side handlers,
- never place private keys in Supabase.

### Real-time behavior
Supabase Realtime can push updates when:
- a proof is accepted,
- a task unlocks,
- an AI policy is updated,
- a submission is verified,
- a public contract event is indexed.

That gives the app live state without fake polling logic.

## 7. Smart contract layer

### Chain
Base Sepolia only.

### Contract set
1. `ProoflyRegistry.sol`
   - stores proof acceptance state,
   - maps wallet address to verified-human status,
   - stores public proof references if needed.

2. `ProoflyPolicy.sol`
   - stores agent policy hashes,
   - stores session limits,
   - stores contract allowlists,
   - stores expiration windows.

3. `ProoflySession.sol`
   - tracks active delegation sessions,
   - allows/rejects actions against policy,
   - can expire sessions cleanly.

4. `ProoflyTaskGate.sol`
   - unlocks protected actions or tasks after verification.

### Contract responsibilities
- verify proof state,
- store canonical trust records,
- enforce permissions,
- expose readable state to the frontend.

### Contract responsibilities it should not take over
- private key custody,
- media uploads,
- app UX state,
- local wallet storage.

## 9. API design

All endpoints should be real and deterministic.

### Wallet session
`POST /api/wallet/challenge`  
Returns a signed challenge for wallet login.

`POST /api/wallet/verify`  
Verifies the signed challenge and starts a session.

### Proof flow
`POST /api/zk/worldid/start`  
Creates a proof request payload.

`POST /api/zk/worldid/confirm`  
Validates the returned proof and writes the result to Supabase and optionally to Base Sepolia.

### AI leash
`POST /api/agent/policy/create`  
Creates a policy record.

`POST /api/agent/policy/authorize`  
Signs or verifies a bounded delegation session.

`POST /api/agent/action/check`  
Checks whether the action is allowed by the current policy.

### Media signing
`POST /api/media/hash`  
Receives metadata for a local file and records the resulting hash.

`POST /api/media/sign`  
Stores the wallet signature and proof metadata.

### On-chain synchronization
`POST /api/chain/event-sync`  
Indexes Base Sepolia events into Supabase.

## 11. End-to-end flows

### Flow A: wallet creation
1. User opens the extension.
2. User creates a wallet locally.
3. Key material is encrypted and stored locally.
4. Address is displayed.
5. User can now connect to the dApp.

### Flow B: human verification
1. User opens the Proofly verification screen.
2. Wallet starts World ID verification.
3. Proof is returned.
4. Wallet stores the proof state.
5. Supabase records the event.
6. Base Sepolia contract is updated if on-chain enforcement is needed.
7. DApp unlocks human-only actions.

### Flow C: AI leash
1. User opens policy screen.
2. User defines spend and contract limits.
3. Policy hash is stored in Supabase and on-chain.
4. AI agent requests a transaction.
5. Wallet checks policy locally.
6. If allowed, the wallet signs.
7. If not allowed, the wallet rejects.

### Flow D: media signing
1. User selects audio or media.
2. Local hash is computed.
3. Wallet signs the hash.
4. Proof state is attached.
5. Supabase stores the event.
6. Recipient can verify the source trail.

## 12. Real-time state handling

To keep the app live and consistent:
- wallet state stays local,
- operational data syncs with Supabase,
- public trust state syncs with Base Sepolia,
- contract events are indexed into Supabase,
- frontend listens to Supabase Realtime for changes.

This creates a real-time system without pretending all state belongs in one place.

---

## 13. Security rules

### Wallet-side rules
- no remote private key access,
- no silent signing,
- no hidden approvals,
- no remote code execution in the extension,
- no proof reuse where nullifiers should prevent reuse.

### Backend rules
- RLS on every table,
- least-privilege service roles,
- edge functions for privileged operations,
- signed webhook verification,
- strict input validation.

### Chain rules
- only Base Sepolia in v1,
- contracts are the public enforcement layer,
- contract addresses must be fixed per deployment,
- tx hashes should be stored for auditability.

## 16. Build rule for the team

Any feature in the product must exist in one of these four places:

1. wallet extension,
2. Base Sepolia contract,
3. Supabase backend,
4. Next.js frontend.

If a flow cannot be traced to one of those, it is not part of the real system.
